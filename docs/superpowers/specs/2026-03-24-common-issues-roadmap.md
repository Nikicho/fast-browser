# Fast-Browser 通用问题清单（2026-03-24）

## 背景

本轮问题整理来自以下真实使用记录：
- `D:\AIWorks\docs\javdb-adapter-dev-log.md`
- `D:\AIWorks\skills\fast-browser\docs\BILIBILI_ADAPTER_DEV_LOG.md`
- `D:\AIWorks\skills\fast-browser\docs\USAGE_AND_ISSUES.md`

目标不是按站点逐个修，而是提炼为 Fast-Browser 在更多真实网站上都会遇到的通用底座问题。

---

## 一、问题分层

### A. 浏览器底座问题

#### 1. 动态页面数据提取仍过度依赖 `evalExpression`

表现：
- `snapshot` 对 SPA / 登录态页面只能看到部分文本或初始 HTML
- 真实业务数据仍需靠 `evalExpression` 从已渲染文本中提取
- adapter 被迫在命令里写大量字符串清洗、分段扫描、正则过滤逻辑

影响：
- adapter 可维护性差
- 同类站点会重复编写文本流抽取逻辑
- 对 agent 来说，“站点探索”仍然偏重

本质：
- Fast-Browser 还缺一个比 `snapshot` 更适合“动态文本流 / feed / card 列表抽取”的中层能力

#### 2. 长列表与延迟加载内容抓取不稳定

表现：
- 默认 `snapshot()` 容易截断列表
- 需要手动滚动、再拉高 `maxItems`
- 对“加载更多”“无限滚动”类页面，抓取策略不统一

影响：
- 每个 adapter 都要自己写滚动 + 等待 + 重新 snapshot 的套路
- 对 feed、榜单、评论页、搜索结果页很不友好

本质：
- 缺少标准化的“采集更多内容”能力

#### 3. 页面前置 gate 缺少通用处理机制

表现：
- 年龄确认 modal
- Cookie 同意弹窗
- 继续浏览 / 接受条款 / 风险确认之类前置交互
- 目前通常写在 adapter 里临时处理

影响：
- 同类逻辑会在多个 adapter 里重复
- 首次进入页面的稳定性差

本质：
- 缺少通用的“页面解锁 / gate 处理”层

#### 4. 多标签页能力不足

表现：
- 当前更适合控制单一活动页面
- 一次性打开多个详情页、对比页、评论页不顺
- 批量探索和并行对比场景受限

影响：
- 某些高效率工作流无法自然表达
- agent 只能回退到“记录 URL，人工稍后自己开”

本质：
- 浏览器 runtime 还缺多 tab 抽象

---

### B. 资产沉淀与校验问题

#### 5. `flow` / `case` 结构校验还不够防错

表现：
- assertion 字段名写错时，容易直到运行阶段才暴露
- `value` / `selector` / `count` 的搭配不够直观
- 真实使用中仍会出现 JSON 结构写对一半、运行时报错的情况

影响：
- 保存资产的体验不稳定
- agent 在沉淀资产时容易留下半成品

本质：
- schema 校验还不够强，错误提示也还不够面向 agent

#### 6. `flow id` / 文件名 / `case` 引用一致性缺少强约束

表现：
- `best-weekly.flow.json` 和 flow 内 `id` 不一致
- case 引用 flow 名称时依赖 agent 自己猜
- 需要再跑 `flow list` 才知道真实可用 id

影响：
- 易出现“文件在，但跑不起来”
- 沉淀链路仍偏脆

本质：
- 保存阶段缺少一致性检查和自动纠偏

---

### C. Workflow / 经验沉淀问题

#### 7. adapter 层仍承担过多站点特定的文本解析脏活

表现：
- 标题清洗
- 时间解析
- 干扰词过滤
- 评分、时长、标志位从文本中硬提取

影响：
- 新站点适配成本偏高
- 同类站点经验难以下沉到共用能力

本质：
- Fast-Browser 目前更擅长“控制页面”，还不够擅长“辅助提取半结构化页面数据”

#### 8. 使用记录与经验文档本身也存在一致性问题

表现：
- 文件名与内容不一致，例如 `BILIBILI_ADAPTER_DEV_LOG.md` 实际写的是 GitHub adapter 使用记录
- 多份日志存在编码问题
- 通用问题和站点问题混写

影响：
- 后续复盘、训练、技能迭代时会误导
- agent 难以把旧记录当成高质量输入

本质：
- 经验沉淀格式还不够统一

---

## 二、优先级建议

### P1：应优先修复

1. 动态页面数据提取中层能力
2. 长列表 / 延迟加载内容采集标准化
3. `flow` / `case` 更强保存前校验
4. `flow id` / 文件名 / case 引用一致性校验

原因：
- 这四项直接影响“能不能更快适配更多网站”
- 也最影响 agent 后续减少 `snapshot/eval` 依赖

### P2：第二批修复

1. 页面 gate 通用处理
2. 多 tab 抽象
3. 半结构化页面文本抽取辅助能力

原因：
- 价值高，但改动面更大
- 适合作为第二阶段底座增强

### P3：流程与文档治理

1. 统一试用日志命名规范
2. 统一 UTF-8 编码
3. 站点问题与通用问题分离记录

原因：
- 不影响 CLI 核心功能
- 但对长期演进质量有帮助

---

## 三、下一步建议

推荐修复顺序：

### 批次 1
- `flow/case` 保存前强校验
- flow id / 文件名 / case 引用一致性检查

### 批次 2
- 标准化长列表抓取能力
- 标准化页面文本块/卡片抽取能力

### 批次 3
- 通用 gate 处理
- 多 tab 支持

---

## 四、结论

当前 Fast-Browser 的主要短板，已经不再是“没有命令”，而是：
- 对动态页面数据的中层抽取能力不足
- 对可复用资产的保存校验还不够强
- 对高频页面模式（feed、榜单、评论、弹窗、详情批量打开）的底座抽象还不够完整

也就是说，下一阶段的重点不该是继续堆更多命令，而应该是：
- 提升页面数据抽取能力
- 提升资产沉淀稳定性
- 提升复杂网页模式的通用支持
